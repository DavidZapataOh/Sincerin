# UltraHonk Verifier Reference Code

This directory contains reference implementations for the UltraHonk verifier,
used as references for implementing the Go version in the Sincerin L1 precompile.

## Sources

### Rust (zkVerify / Horizen Labs)
- Primary: https://github.com/miquelcabot/ultrahonk_verifier
- Related: https://github.com/zkVerify/ultraplonk_verifier (UltraPlonk predecessor)

### Solidity (Aztec / Barretenberg)
- Generated by bb (Barretenberg) for on-chain verification

## Rust Reference Files

- `constants.rs` - Protocol constants (CONST_PROOF_SIZE_LOG_N=28, NUMBER_OF_ENTITIES=40, etc.)
- `proof.rs` - Proof deserialization structs (PlainProof, ZKProof, G1ProofPoint) and field order
- `key.rs` - Verification key struct and deserialization (27 G1 commitments + 4 u64 params)
- `lib.rs` - Main verify() function, sumcheck verification, shplemini verification, pairing check
- `transcript.rs` - Fiat-Shamir transcript (Keccak256-based challenge generation) -- COMPLETE
- `srs.rs` - Fixed G2 SRS points for pairing verification
- `types.rs` - Type aliases (Fr, Fq, G1, G2, Bn254)
- `utils.rs` - Serialization utilities (read_g1, read_g2, read_u64, BigEndian conversions)
- `commitment.rs` - Gemini fold evaluation computation (compute_squares, compute_fold_pos_evaluations)
- `relations.rs` - All 26 UltraHonk relation evaluations (arithmetic, permutation, lookup, etc.)
- `Cargo.toml` - Dependencies (ark-bn254, sha3/keccak256, etc.)

## Solidity Reference Files

- `BaseHonkVerifier.sol` - Main verifier contract (verify, verifySumcheck, verifyShplemini)
- `Transcript.sol` - Fiat-Shamir transcript (Keccak256 challenges)
- `HonkTypes.sol` - Proof struct, VK struct, transcript struct
- `Relations.sol` - All 26 subrelation evaluations
- `CommitmentScheme.sol` - KZG / Shplemini commitment scheme
- `Fr.sol` - Field arithmetic library
- `Errors.sol` - Error definitions
- `utils.sol` - Utility functions

## Key Architecture Decisions

1. **Proof point format**: G1ProofPoint uses split representation (x_0, x_1, y_0, y_1) where
   x = x_0 | (x_1 << 136), same in both Solidity and Rust
2. **Transcript**: Keccak256 with big-endian 32-byte encoding of field elements
3. **Challenge splitting**: Challenges are split into 128-bit halves via split_challenge()
   - lower = limbs[0] | (limbs[1] << 64)
   - upper = limbs[2] | (limbs[3] << 64)
4. **SRS G2 points**: Hardcoded (same as Barretenberg/Aztec)
5. **Proof types**: Both Plain and ZK supported (ZK adds libra commitments + masking)
6. **VK format**: 1760 bytes (4 u64 params big-endian + 27 G1 points at 64 bytes each)
7. **CONST_PROOF_SIZE_LOG_N** = 28 (maximum supported circuit size = 2^28)
8. **Deserialization order**: w4 comes AFTER lookup_read_counts and lookup_read_tags (not after w3!)
9. **VK G1 points**: Standard affine (x, y) each 32 bytes big-endian (NOT split like proof points!)
10. **Verification flow**: generate_transcript -> verify_sumcheck -> verify_shplemini -> pairing_check
11. **Pairing check**: e(P_0, SRS_G2) * e(-kzg_quotient, SRS_G2_VK) == 1
12. **NUMBER_OF_ENTITIES**: 40 (35 unshifted + 5 shifted)
13. **NUMBER_OF_SUBRELATIONS**: 26 (batched with 25 alpha challenges)
14. **pub_inputs_offset**: Hardcoded to 1 in the Rust verifier (not from VK)
